# 🗺️ 개발 로드맵 (ROADMAP)

## 📌 프로젝트 개요
Notion을 CMS로 활용한 개인 기술 블로그 플랫폼 구축 프로젝트입니다.

---

## 🎯 전체 개발 계획

```
Phase 1: 프로젝트 골격 (구조, 환경 설정)
    ↓
Phase 2: 공통 모듈 (모든 기능에서 쓰는 것들)
    ↓
Phase 3: 핵심 기능 (가장 중요한 기능)
    ↓
Phase 4: 추가 기능 (부가적인 기능)
    ↓
Phase 5: 최적화 및 배포
```

---

## 📍 Phase 1: 프로젝트 골격 (구조, 환경 설정)

### ⏱️ 예상 소요 시간
**1-2일**

### 🤔 왜 이 순서인가?

**가장 기초가 되는 단계입니다.**

- 견고한 프로젝트 골격이 없으면 이후 모든 기능 개발이 어렵습니다
- Notion API 환경이 설정되지 않으면 다른 기능을 개발할 수 없습니다
- 폴더 구조와 설정이 명확해야 팀 작업이나 유지보수가 수월합니다
- **이 단계를 완료하지 않으면 다른 모든 단계는 불가능합니다**

### 📋 작업 내용

#### Next.js 프로젝트 구조 설정
- [ ] Next.js 15.5.3 앱 라우터 기본 구조 확인
- [ ] TypeScript 설정 확인
- [ ] Tailwind CSS v4 설정 확인
- [ ] shadcn/ui 초기 설정 확인
- [ ] ESLint + Prettier 설정 확인
- [ ] Husky + lint-staged 설정 확인

#### Notion API 연동 환경 구축
- [ ] `.env.local` 파일 생성
- [ ] Notion API 키 설정
- [ ] Notion 데이터베이스 ID 설정
- [ ] `@notionhq/client` 패키지 설치 및 확인
- [ ] Notion API 클라이언트 초기화 코드 작성
- [ ] API 기본 테스트 (데이터 조회 확인)

#### 기본 레이아웃 구조 생성
- [ ] 프로젝트 디렉토리 구조 설계
  - `/app` - Next.js 페이지 및 라우트
  - `/components` - React 컴포넌트
  - `/lib` - 유틸리티, 타입, API 클라이언트
  - `/public` - 정적 파일
- [ ] 기본 레이아웃 컴포넌트 뼈대 생성
- [ ] 페이지 라우트 기본 구조 생성

### ✅ 완료 기준
- [ ] Notion API 클라이언트가 정상 작동하고 데이터 조회 가능
- [ ] 환경 변수가 제대로 설정됨
- [ ] 프로젝트 폴더 구조가 명확하고 정리됨
- [ ] 개발 서버가 정상 실행됨 (`npm run dev`)

## 📍 Phase 2: 공통 모듈 (모든 기능에서 쓰는 것들)

### ⏱️ 예상 소요 시간
**2-3일**

### 🤔 왜 이 순서인가?

**각 기능 개발의 기초가 되는 단계입니다.**

- 모든 기능에서 반복적으로 사용할 함수와 컴포넌트를 먼저 만들어야 합니다
- 공통 모듈이 없으면 각 기능마다 중복된 코드를 작성하게 되어 유지보수가 어렵습니다
- 타입 정의를 먼저 하면 다른 기능 개발 시 타입 안정성을 보장할 수 있습니다
- **Phase 1이 완료된 직후에 Phase 2를 해야, Phase 3 이후 개발이 빨라집니다**

### 📋 작업 내용

#### Notion API 공통 함수
- [ ] `fetchPages()` - Published 글 목록 조회
  - 데이터베이스에서 Published 상태의 글만 조회
  - 최신순(역순) 정렬
  - 페이지 크기 제한 처리
- [ ] `fetchPageContent()` - 특정 글의 상세 콘텐츠 조회
  - slug 기반 글 조회
  - 메타데이터(제목, 카테고리, 태그, 발행일) 포함
- [ ] `fetchPageBlocks()` - Notion 블록 데이터 조회
  - 페이지의 모든 블록 조회
  - HTML/마크다운 변환 준비
- [ ] `fetchCategories()` - 카테고리 목록 조회
  - 모든 카테고리 옵션 조회

#### 공통 컴포넌트
- [ ] `Header` 컴포넌트
  - 로고/사이트 제목
  - 네비게이션 메뉴
  - 검색창 자리 (Phase 4에서 구현)
- [ ] `Footer` 컴포넌트
  - 저작권 정보
  - 링크 정보
- [ ] `Card` 컴포넌트
  - 글 목록용 카드
  - 제목, 카테고리, 발행일, 썸네일 표시

#### 공통 타입 정의
- [ ] `Post` 타입 (글 정보)
  ```typescript
  - id: string
  - slug: string
  - title: string
  - category: string
  - tags: string[]
  - publishedDate: Date
  - content: string
  - thumbnail?: string
  ```
- [ ] `Category` 타입 (카테고리 정보)
- [ ] `Block` 타입 (Notion 블록 정보)

### ✅ 완료 기준
- [ ] 모든 공통 함수가 TypeScript 타입과 함께 정의됨
- [ ] 공통 컴포넌트가 스타일링 완료되어 재사용 가능
- [ ] 타입 정의가 완료되고 TypeScript 에러 없음
- [ ] 공통 함수 테스트 완료 (콘솔로 출력하여 데이터 확인)

## 📍 Phase 3: 핵심 기능 (가장 중요한 기능)

### ⏱️ 예상 소요 시간
**3-4일**

### 🤔 왜 이 순서인가?

**블로그로서 가장 중요한 기능입니다.**

- 이 단계에서 구현할 글 목록과 상세 페이지가 없으면 블로그가 성립하지 않습니다
- 다른 모든 기능(검색, 필터링 등)은 이 핵심 기능을 보완하는 역할입니다
- Phase 1과 2가 완료되었으므로, 이제 비로소 사용자가 볼 수 있는 화면을 만들 수 있습니다
- **이 단계를 완료하면 블로그의 최소 기능(MVP)이 완성됩니다**

### 📋 작업 내용

#### 블로그 글 목록 페이지 (`/`)
- [ ] `fetchPages()` 함수 활용하여 글 목록 데이터 조회
- [ ] Server Actions로 데이터 페칭 구현
- [ ] `Card` 컴포넌트로 글 목록 렌더링
- [ ] 글 카드 클릭 시 상세 페이지로 이동
- [ ] 반응형 그리드 레이아웃 (모바일/태블릿/데스크톱)
  - 모바일: 1열
  - 태블릿: 2열
  - 데스크톱: 3열

#### 블로그 글 상세 페이지 (`/posts/[slug]`)
- [ ] 동적 라우트 구현 (`[slug]` 페이지)
- [ ] URL slug로 해당 글 조회
- [ ] `fetchPageContent()` 함수로 글의 메타데이터 조회
- [ ] Notion 블록을 HTML/마크다운으로 렌더링
  - 텍스트 블록 렌더링
  - 헤딩 렌더링
  - 리스트 렌더링
  - 코드 블록 렌더링
  - 이미지 렌더링
- [ ] 메타데이터 표시
  - 발행일
  - 카테고리
  - 태그

#### Notion 컨텐츠 렌더링
- [ ] `notion-to-md` 또는 유사 라이브러리 선택 및 설치
- [ ] Notion 블록을 마크다운으로 변환
- [ ] 마크다운을 React 컴포넌트로 렌더링 (`react-markdown` 등)
- [ ] 코드 문법 강조 설정 (Prism.js 또는 유사)

### ✅ 완료 기준
- [ ] 글 목록 페이지가 정상적으로 표시됨
- [ ] 글 제목을 클릭하면 상세 페이지로 이동됨
- [ ] 상세 페이지의 Notion 콘텐츠가 제대로 렌더링됨
- [ ] 메타데이터(발행일, 카테고리, 태그)가 정상 표시됨
- [ ] 모든 화면 크기에서 반응형으로 표시됨
- [ ] 존재하지 않는 글 접근 시 404 페이지 표시

## 📍 Phase 4: 추가 기능 (부가적인 기능)

### ⏱️ 예상 소요 시간
**2-3일**

### 🤔 왜 이 순서인가?

**핵심 기능을 강화하는 단계입니다.**

- Phase 3까지 완료하면 이미 블로그의 기본 기능이 작동합니다
- 추가 기능(검색, 필터링, SEO)은 "있으면 좋지만 없어도 작동하는" 기능입니다
- 핵심 기능이 완성된 후에 추가하면, 기능 간 의존성 문제 없이 깔끔하게 개발할 수 있습니다
- **이 단계에서 사용자 경험을 크게 향상시킬 수 있습니다**

### 📋 작업 내용

#### 카테고리 필터링
- [ ] Sidebar에 카테고리 목록 표시
- [ ] 카테고리 선택 UI 구현
- [ ] 선택한 카테고리로 글 필터링
- [ ] 활성 카테고리 하이라이트
- [ ] URL 쿼리 파라미터(`?category=react`) 활용하여 상태 관리
- [ ] `/category/[name]` 페이지 구현 (선택사항)

#### 검색 기능
- [ ] 헤더에 검색 입력창 추가
- [ ] 제목 및 태그 기반 검색 로직 구현
- [ ] 검색 결과 실시간 표시
- [ ] 검색 결과가 없을 때 안내 메시지 표시
- [ ] 검색어 하이라이트 (선택사항)

#### SEO 최적화
- [ ] 각 페이지별 메타 태그 설정
  - `title` - 페이지 제목
  - `description` - 페이지 설명
  - `og:image` - 썸네일 이미지
  - `og:url` - 페이지 URL
- [ ] `sitemap.xml` 생성 및 설정
- [ ] `robots.txt` 생성
- [ ] 구조화된 데이터 (Schema.org) 추가

### ✅ 완료 기준
- [ ] 카테고리 필터링이 정상 작동
- [ ] 검색 기능이 정상 작동
- [ ] 필터와 검색 상태가 URL에 반영됨
- [ ] 메타 태그가 모든 페이지에 설정됨
- [ ] `sitemap.xml`과 `robots.txt`가 생성됨

## 📝 개발 가이드

### 왜 순서가 중요한가?

```
🔴 잘못된 순서 예시:
Phase 3 → Phase 1 → Phase 2 
❌ 결과: 환경이 없어서 기능 개발 불가능, 중복 코드 양산

✅ 올바른 순서:
Phase 1 → Phase 2 → Phase 3 → Phase 4 → Phase 5
✅ 결과: 각 단계가 다음 단계의 기초가 되어 효율적 개발
```

### 각 Phase 완료 체크리스트
```
[ ] 작업 내용 완료
[ ] 완료 기준 모두 충족
[ ] 코드 리뷰 완료
[ ] 테스트 완료
[ ] ROADMAP.md 진행상황 업데이트
[ ] 커밋 메시지 작성
```

### 커밋 메시지 규칙
- Phase 1: `feat: Phase 1 프로젝트 골격 완료`
- Phase 2: `feat: Phase 2 공통 모듈 완료`
- Phase 3: `feat: Phase 3 핵심 기능 완료 (MVP)`
- Phase 4: `feat: Phase 4 추가 기능 완료`
- Phase 5: `feat: Phase 5 최적화 및 배포 완료`

### 핵심 원칙
1. **순서 준수**: 절대 순서를 바꾸지 말 것. 각 Phase는 이전 Phase에 의존합니다
2. **완료 기준 확인**: 체크리스트의 모든 항목을 완료한 후 다음 단계로 진행
3. **문제 해결**: 문제가 발생하면 이전 Phase로 돌아가 원인을 파악하고 해결
4. **진행상황 업데이트**: 각 Phase 완료 후 ROADMAP.md의 상태 업데이트
